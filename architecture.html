express233 架构流程

CI 是推模型：开发者把版本包推到中央控制台。CD 是拉模型：每台服务器只向中央拉取自己被允许拿到的、已经替换稳定的最终内容。

CI push render linux windows canary

开发 / CI

push model

版本包上传

CI 产出二进制、配置模板、脚本、静态资源，然后推送到中央。

  • 版本：1.0.1、1.1.0、hotfix
  • 平台标签：all、linux-amd64、windows-amd64
  • 状态：draft → review → published

只负责交付原始模板

CI 不关心每台服务器的 server_id，也不在节点侧做替换。

config/application.yaml
bin/game-server-linux-amd64
scripts/restart.sh

中央控制台

stable render

权限与内容选择中心

中央决定每台服务器能拉到什么文件，而不是节点自由选择。

  • 按 project / version / server_id 生成清单
  • 按 os / arch / tag 过滤文件
  • 按 token、团队权限、发布状态控制下载

中央式模板替换

server.yaml replacements 在中央完成，节点拿到的是已经渲染好的最终配置。

server_id: game-logic-01
application.yaml:
  game.serverId: game-logic-01
  mysql.url: jdbc:mysql://db-a/game

稳定性边界

替换逻辑只有中央一份实现。所有节点拉取结果可预览、可审计、可复现。

游戏服务器 / CD

pull model
game-logic-01 linux-amd64，只拉 all + linux-amd64；配置已替换为 logic-01。
game-logic-02 windows-amd64，只拉 all + windows-amd64；不会拿到 Linux 二进制。
canary-03 额外 tag=canary，只拉允许的灰度文件和稳定渲染配置。

安全部署

节点拉取到临时目录，再 stop → swap → start，保留 logs/run。

pull to .tmp/server_id
stop old process
rsync swap
run scripts/restart.sh
1. CI 推送版本 构建产物与模板上传中央，附带平台标签和版本状态。
2. 中央发布与审计 发布前检查 server.yaml、配置 basename、文件标签与权限。
3. 中央渲染最终配置 按 server_id 在中央替换 JSON / YAML / properties。
4. CD 节点拉取 服务器只拉自己可见的最终内容，部署过程幂等安全。
核心差异

CI 推模型解决“版本从哪里来”；CD 拉模型解决“节点何时、拉什么、怎么安全切换”。

为什么中央替换

所有模板替换集中在中央，避免每台服务器脚本版本不同、环境不同导致结果漂移。

为什么可控

中央用 project、version、server_id、token、tag 共同决定下载清单，每台服务器拿到的内容可解释、可追踪。
